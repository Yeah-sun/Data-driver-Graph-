# 页面数据流审视

## 设计原则

当前 UI 原型里还有不少本地 `mock.js` 数据。正式联调时，除纯路由跳转、前端折叠、图谱布局重排、抽屉开关这类 UI 状态外，页面上能被用户看到的业务数据都应来自接口。

空间使用原则：

- 首页不做欢迎页，只展示进度、指标、任务和下一步动作。
- 每个卡片必须对应一个接口数据源或一个明确的用户操作。
- 列表、指标、状态、图谱、计划文件都必须动态请求。
- 本地 mock 只作为接口失败时的演示降级，不作为正式数据来源。
- 管理员页面只展示管理数据，不进入学生/求职者业务功能。

## 全局框架

区域职责：

- 左侧导航：根据登录角色动态返回可访问菜单。
- 顶部搜索：按角色搜索业务对象，学生搜岗位/技能/图谱节点，管理员搜用户/导入任务/岗位版本。
- 顶部通知：展示当前用户的待处理事项和后台任务摘要。
- 用户信息：展示当前登录用户、角色、学校或组织。
- GitHub 源码卡片：静态外链，不请求业务接口。

动态请求：

| 区域 | 获取接口 | 说明 |
| --- | --- | --- |
| 会话恢复 | `GET /api/auth/me` | 刷新页面后恢复用户信息和角色 |
| 导航权限 | `GET /api/auth/me` 或 `GET /api/navigation` | 第一版可从角色本地推导，正式版建议后端返回 |
| 顶部通知 | `GET /api/notifications` | 返回待复核、任务运行、评测完成等消息 |
| 最近任务 | `GET /api/tasks/recent` | 工作台和顶部状态可复用 |

不应占用空间：

- 不要固定展示无意义欢迎语。
- 不要放静态装饰卡片。
- 通知为空时只显示轻量空态，不保留大面积空白。

## 登录 / 注册

页面部分：

| 部分 | 作用 | 获取数据 | 提交数据 |
| --- | --- | --- | --- |
| 登录表单 | 用户进入系统 | 可选 `GET /api/auth/options` | `POST /api/auth/login` |
| 注册表单 | 创建账号 | 可选学校/角色配置 | `POST /api/auth/register` |
| 游客演示 | 比赛快速演示 | 无 | `POST /api/auth/demo-session` |

显示字段：

- 学校列表、角色列表、账号、密码、错误提示、登录模式。

空间建议：

- 左侧品牌说明只保留系统定位和 3 个核心指标。
- 右侧表单保持紧凑，不放多余营销文案。

## 我的工作台

页面部分：

| 部分 | 作用 | 获取接口 | 提交/动作 |
| --- | --- | --- | --- |
| 下一步建议 | 告诉用户当前该做什么 | `GET /api/dashboard/summary` | 跳转到 `workflow.nextPath` |
| 统计卡片 | 展示当前业务指标 | `GET /api/dashboard/summary` | 无 |
| 流程轨迹 | 展示简历、岗位、匹配、路径进度 | `GET /api/dashboard/summary` | 点击节点可跳转 |
| 快速动作 | 进入关键页面 | `GET /api/dashboard/summary` | 路由跳转 |
| 后台任务 | 展示异步任务进度 | `GET /api/tasks/recent` | 可刷新 |

必须动态显示：

- 当前匹配分、推荐岗位数、待复核技能数、测试样本数。
- 流程进度和每个阶段状态。
- 快速动作列表应由后端根据当前阶段生成，不能写死。

空间建议：

- 工作台保留 1 行建议、1 行指标、下方 2 到 3 个工作卡片。
- 没有数据时显示短空态，不保留大卡片空白。

## 简历中心

页面部分：

| 部分 | 作用 | 获取接口 | 提交/动作 |
| --- | --- | --- | --- |
| 文件上传 | 上传 PDF/DOCX | `GET /api/resumes` | `POST /api/resumes/upload` |
| 解析步骤 | 展示解析任务状态 | `GET /api/tasks/{taskId}` | `POST /api/resumes/{resumeId}/parse` |
| 待复核项 | 人工确认低置信技能 | `GET /api/resumes/{resumeId}/profile` | `PUT /api/resumes/{resumeId}/profile` |
| 技能画像 | 展示技能和熟练度 | `GET /api/resumes/{resumeId}/profile` | `PUT /api/resumes/{resumeId}/profile` |

必须动态显示：

- 上传文件列表、文件状态、解析步骤、技能标签、置信度、复核原因。

空间建议：

- 文件列表只展示最近 2 到 5 条，历史简历放“查看全部”。
- 复核项没有时收起该区，避免空白。
- 技能画像应优先展示高价值技能，低权重技能折叠。

## 岗位探索

页面部分：

| 部分 | 作用 | 获取接口 | 提交/动作 |
| --- | --- | --- | --- |
| 筛选条 | 按赛道/级别/时间筛选 | `GET /api/jobs/filters` | 触发 `GET /api/jobs` |
| 岗位列表 | 浏览候选岗位 | `GET /api/jobs` | 点击岗位获取详情 |
| 岗位详情 | 辅助决策 | `GET /api/jobs/{jobId}` | `POST /api/jobs/target` |
| 岗位对比 | 比较岗位差异 | `POST /api/jobs/compare` | 获取对比结果 |

必须动态显示：

- 岗位热度、匹配度、更新时间、必备技能、加分技能、职责摘要。

空间建议：

- 左侧列表滚动，右侧详情固定，避免整页浪费。
- 没选岗位时右侧显示“请选择岗位”的紧凑空态。

## 匹配分析

页面部分：

| 部分 | 作用 | 获取接口 | 提交/动作 |
| --- | --- | --- | --- |
| 综合评分 | 快速判断匹配结果 | `GET /api/matching/reports/{reportId}` | 无 |
| 分项解释 | 解释评分来源 | `GET /api/matching/reports/{reportId}` | 可重新匹配 |
| 命中与缺口 | 展示技能差距 | `GET /api/matching/reports/{reportId}` | 跳转学习路径 |
| 系统结论 | 给出下一步建议 | `GET /api/matching/reports/{reportId}` | 无 |

必须动态显示：

- 综合分、等级、必备技能命中、加分技能命中、经验匹配、语义相似、差距技能、解释文本。

空间建议：

- 分数和解释不能占满大面积，应该让差距技能和下一步动作更突出。
- 如果没有报告，页面自动发起 `POST /api/matching/tasks` 或提示先选择岗位。

| 部分 | 作用 | 获取接口 | 提交/动作 |

- 未生成计划前只显示请求面板，不预留空白时间线。

## 能力图谱

页面部分：

| 部分 | 作用 | 获取接口 | 提交/动作 |
| --- | --- | --- | --- |
| 图谱画布 | 渲染 Neo4j 节点关系 | `GET /api/graph/neo4j?job=...` | 前端拖拽/缩放/重排 |
| 节点说明 | 展示选中节点属性 | `GET /api/graph/nodes/{nodeId}` | 点击节点触发 |
| 展开子技能 | 局部扩展图谱 | `GET /api/graph/nodes/{nodeId}/neighbors` | 点击展开 |
| 版本演化 | 展示岗位能力变迁 | `GET /api/graph/versions?job=...` | 无 |

必须动态显示：

- 节点、关系、关系权重、节点类型、摘要、可展开数量、版本时间线。

空间建议：

- 图谱画布是主区域，占最大空间。
- 节点说明和版本演化是辅助区，不应比画布更抢空间。
- 图谱节点过多时必须分页或局部展开，不能一次请求全库。

页面部分：

- 不要在后台放学生视角功能入口。
- 指标面板只展示核心评测项，详细报告通过报告页或下载查看。

## Mock 替换优先级

第一优先级：

- `Auth API`
- `Dashboard API`
- `Resume API`
- `Job API`
- `Matching API`

第二优先级：

- `Learning API`
- `Graph API`
- `Task API`

第三优先级：

- `Admin API`
- `AI Agent API`

验收标准：

- 每个页面刷新后都能从接口恢复当前状态。
- 每个业务卡片都有明确接口来源。
- 每个按钮要么请求接口，要么是纯前端 UI 行为，要么是路由跳转，不能含糊。
- 页面没有真实数据时显示空态，不展示假数据。
- 接口失败时允许 mock fallback，但必须有错误提示。
